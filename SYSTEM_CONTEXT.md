## 1. Project Overview

**Project name:** DocTalk

**What problem it solves:** DocTalk is a RAG (Retrieval-Augmented Generation) web application. Users upload a PDF document, the backend extracts and chunks its text, generates vector embeddings for each chunk, and stores everything in MongoDB. Users can then ask natural-language questions about the PDF and receive answers generated by an LLM, grounded in the relevant chunks retrieved via vector similarity search.

**Main features:**

- User authentication (signup, login, JWT-based session)
- PDF upload and text extraction
- Automatic text chunking with configurable size and overlap
- Vector embedding generation via Google Gemini
- Vector similarity search via MongoDB Atlas Vector Search
- Conversation-based Q&A with streaming LLM responses (SSE)
- Source citations (page-level references) returned with every answer

**Current development stage:** Backend API is functional end-to-end. All core pipeline features (auth, upload, chunking, embedding, retrieval, LLM answering with streaming) are verified working. Document management (list, get, rename, star, trash, restore, permanent delete) and GridFS file storage (PDF persistence, download, cleanup) are implemented. Frontend is built separately via Lovable and is not yet connected to this backend.

**What is already working:**

- Health check endpoint (`GET /health` via `health.routes.js`)
- User signup with email validation and password hashing
- User login with JWT generation
- Get current user profile (protected route)
- PDF upload: multer parses file, pdf-parse extracts text, text is chunked, chunks are saved to MongoDB, Gemini embeddings are generated in batches and stored
- GridFS storage: raw PDF bytes are persisted to a `pdfs` GridFS bucket during upload
- Document management: list (with filter: recent/starred/trash), get, rename, star/unstar, trash, restore, permanent delete (with GridFS cleanup)
- PDF download: stream original PDF from GridFS
- Conversation creation tied to a document
- Conversation listing (sorted by most recent)
- Get conversation with full message history
- Send message: question is embedded, vector search retrieves top matching chunks, Groq streams the answer via SSE, full answer and citations are saved
- Centralized error handling middleware
- CORS middleware configured (permissive, no origin restriction)
- `.env.example` with all required variable placeholders

**What is incomplete / pending:**

- Frontend wiring (Lovable frontend exists separately, not connected)
- Google OAuth
- Multi-document conversation support
- Proper README

### Request Flow — Document Upload

```
POST /api/documents (multipart/form-data, field: "file")
    │
    ├─ auth.middleware.js → verifies JWT, sets req.user
    ├─ upload.middleware.js → multer parses PDF into req.file.buffer
    │
    ▼
document.controller.js: uploadDocument()
    │
    ├─ Creates Document record with status: "processing"
    ├─ Streams raw PDF to GridFS bucket "pdfs"
    │   └─ Captures upload stream ID → document.gridfsFileId
    ├─ pdf-parse extracts text from req.file.buffer
    ├─ Splits text by form-feed (\f) into pages
    ├─ For each page: cleanText() → chunkText() → creates chunk objects
    ├─ Chunk.insertMany() saves all chunks to MongoDB
    ├─ Updates document.pageCount
    ├─ embedChunksForDocument(documentId)
    │       │
    │       ├─ Finds all chunks for this document
    │       ├─ Batches of 8 chunks at a time
    │       ├─ For each chunk: calls Gemini API → gets 768-dim vector
    │       ├─ Updates each chunk with its embedding
    │       ├─ 1 second delay between batches (rate limit protection)
    │       └─ Throws if ALL chunks fail to embed
    │
    ├─ Sets document.status = "ready"
    └─ Returns { documentId, title, chunkCount, status }
```

### Request Flow — Chat / Q&A

```
POST /api/conversations/:id/messages (JSON: { question })
    │
    ├─ auth.middleware.js → verifies JWT, sets req.user
    │
    ▼
conversation.controller.js: sendMessage()
    │
    ├─ Verifies conversation exists and belongs to user
    ├─ Saves user message to messages collection
    ├─ embedText(question) → Gemini → 768-dim question embedding
    ├─ searchChunks(documentId, questionEmbedding)
    │       │
    │       ├─ $vectorSearch aggregation on chunks collection
    │       ├─ Filtered by documentId
    │       ├─ topK=5, numCandidates=100
    │       ├─ Score threshold: 0.75
    │       └─ Deduplication (removes near-identical chunks)
    │
    ├─ Sets SSE headers (Content-Type: text/event-stream)
    ├─ streamAnswer(question, matchedChunks) → Groq streaming
    │       │
    │       ├─ Builds prompt: system instructions + context chunks + question
    │       ├─ Calls Groq llama-3.3-70b-versatile with stream: true
    │       └─ Yields text tokens as they arrive
    │
    ├─ Writes each token to response as SSE event
    ├─ Saves complete assistant message + citations to messages collection
    ├─ Updates conversation.updatedAt
    └─ Sends { done: true, matchedChunks } final event
```

### Important Design Decisions

1. **Layered architecture:** Routes → Controllers → Services → Models. Controllers are thin (parse request, call service, shape response). Business logic lives in services.
2. **Synchronous embedding during upload:** Embeddings are generated inside the upload request (not queued). This simplifies the flow but means large PDFs will have slow upload responses.
3. **Streaming via SSE:** LLM responses stream token-by-token using Server-Sent Events, not waiting for the full completion.
4. **GridFS for file persistence:** PDFs are stored in a `pdfs` GridFS bucket. The raw buffer is streamed to GridFS during upload, before text extraction. GridFS cleanup is best-effort during permanent delete (logged but doesn't block deletion).
5. **Application-level authorization:** MongoDB has no row-level security. Every query on user-owned data must explicitly filter by `userId` in application code.
6. **`dotenv.config()` must be the first line in server.js** before any other `require()`. Modules that read `process.env` at import time will get `undefined` if dotenv hasn't loaded yet.
7. **pdf-parse pinned to v1.1.1:** v2.x completely changed its export shape (exports a class, not a callable function). Do not upgrade without rewriting extraction code.
8. **Centralized error handling:** `error.middleware.js` is mounted last in `server.js`. Errors can be thrown/next'd to be caught centrally, or handled inline with try/catch per controller function.

### Folder Descriptions

**`backend/src/config/`** — External service client initialization. Each file creates and exports a single client instance that reads from `process.env`. These are required at import time, which is why `dotenv.config()` must execute first in `server.js`. Exception: `gridfs.js` exports a lazy-init factory function (not an instance) because it depends on the mongoose connection being established first.

**`backend/src/controllers/`** — Request handlers. Each controller function receives `(req, res)`, parses the request, calls the appropriate service(s), and sends the response. Controllers do not contain business logic beyond request/response transformation.

**`backend/src/middleware/`** — Express middleware. `auth.middleware.js` verifies JWT tokens and attaches `req.user`. `upload.middleware.js` configures multer for PDF-only uploads with a 20MB limit using memory storage. `error.middleware.js` is a centralized 4-arg error handler mounted last in the middleware stack.

**`backend/src/models/`** — Mongoose schemas/models. Define the shape of documents stored in MongoDB collections. Each file exports a single Mongoose model.

**`backend/src/routes/`** — Express router definitions. Each file maps HTTP methods + paths to controller functions, applying middleware as needed. `health.routes.js` handles the `GET /health` endpoint.

**`backend/src/services/`** — Business logic layer. Each service encapsulates a specific concern: authentication operations, text chunking, embedding generation, LLM interaction, or vector search retrieval. Services are pure functions that accept inputs and return results; they do not handle HTTP concerns.

### File: `backend/src/server.js`

**Purpose:** Application entry point. Loads environment variables, initializes Express, connects to MongoDB, mounts route handlers, and starts the HTTP server.

**Dependencies:**

- `dotenv` — Must be loaded before all other imports
- `express` — Web framework
- `cors` — CORS middleware

- `./routes/auth.routes` — Auth routes
- `./routes/document.routes` — Document routes
- `./routes/health.routes` — Health check route
- `./middleware/error.middleware` — Centralized error handler

**Exports:** `app` (Express application instance)

**Key details:**

- `dotenv.config()` is the literal first line before any other `require()`
- Registers `cors()` middleware (permissive, no origin restriction)
- Registers `express.json()` middleware for JSON body parsing
- Health check moved to `health.routes.js`, mounted at `/health`
- Mounts routes: `/api/auth`, `/api/documents`, `/api/conversations`
- General rate limiter on `/api` prefix (10000 max in test, 100 in prod)
- `errorHandler` mounted as the LAST `app.use()` — catches errors from all routes
- `connectDB()` is awaited inside an async `start()` function — server does NOT listen until DB connection succeeds
- On DB connection failure: logs error and calls `process.exit(1)`
- `start()` only called when `require.main === module` — allows `app` to be imported in tests without starting the server

### File: `backend/src/controllers/auth.controller.js`

**Purpose:** Handles user registration, login, and profile retrieval.

**Dependencies:**

- `../models/User` — User model for database queries
- `../services/auth.service` — Password hashing, comparison, token generation

**Exports:** `{ signup, login, logout, googleCallback, me }`

**Constants:**

- `EMAIL_REGEX` = `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` — basic email validation

**Function: `signup(req, res)`**

- Parameters: `req.body` = `{ name, email, password }`
- Returns: `201` with `{ token, user: { id, name, email, createdAt } }`
- Flow:
  1. Validates all fields are present
  2. Validates email format with regex
  3. Validates password length >= 6
  4. Checks for existing user with same email → `409` if found
  5. Hashes password via `hashPassword()`
  6. Creates user in DB
  7. Generates JWT via `generateToken()`
  8. Returns token and user info (never returns `passwordHash`)

**Function: `login(req, res)`**

- Parameters: `req.body` = `{ email, password }`
- Returns: `200` with `{ token, user: { id, name, email } }`
- Flow:
  1. Validates both fields present
  2. Finds user by email → `401` if not found
  3. Compares password via `comparePassword()` → `401` if no match
  4. Generates and returns JWT

**Function: `me(req, res)`**

- Parameters: `req.user.id` (set by auth middleware)
- Returns: `200` with `{ id, name, email, createdAt }`
- Flow:
  1. Finds user by ID, excludes `passwordHash` via `.select("-passwordHash")`
  2. Returns `404` if not found

**Function: `logout(req, res)`**

- Parameters: `req.user` (set by auth middleware, contains `jti` and `exp`)
- Returns: `200` with `{ message: "Logged out successfully" }`
- Flow:
  1. Extracts `jti` and `exp` from decoded JWT
  2. Calculates `expiresAt` from `exp` claim
  3. Creates `TokenBlocklist` record with `jti` and `expiresAt`
  4. Subsequent requests with this token will be rejected by `auth.middleware.js`

**Function: `googleCallback(req, res)`**

- Parameters: `req.user` (set by Passport.js Google strategy)
- Returns: `200` with `{ token, user: { id, email } }`
- Flow:
  1. Generates JWT for the authenticated Google user
  2. Returns token and basic user info

### File: `backend/src/controllers/conversation.controller.js`

**Purpose:** Handles conversation lifecycle and message sending with streaming LLM responses.

**Dependencies:**

- `../models/Conversation` — Conversation model
- `../models/Message` — Message model
- `../models/Document` — Document model (for ownership verification)
- `../services/embedding.service` — `embedText` function
- `../services/retrieval.service` — `searchChunks` function
- `../services/llm.service` — `streamAnswer` function, `NO_CONTEXT_ANSWER` constant (imported but unused in this file; `NO_CONTEXT_ANSWER` is only used inside `llm.service.js` itself)

**Exports:** `{ createConversation, getConversation, listConversations, sendMessage }`

**Function: `createConversation(req, res)`**

- Parameters: `req.body` = `{ documentId }`, `req.user.id`
- Returns: `201` with conversation object
- Flow:
  1. Validates `documentId` is present
  2. Verifies document exists AND belongs to the user (`Document.findOne({ _id, userId })`)
  3. Creates conversation with `documentIds: [documentId]`
  4. Returns conversation metadata

**Function: `getConversation(req, res)`**

- Parameters: `req.params.id`, `req.user.id`
- Returns: `200` with `{ conversation, messages }`
- Flow:
  1. Finds conversation by ID AND userId (ownership check)
  2. Fetches all messages for that conversation, sorted by `createdAt` ascending
  3. Returns `404` if conversation not found

**Function: `listConversations(req, res)`**

- Parameters: `req.user.id`
- Returns: `200` with array of conversations sorted by `updatedAt` descending

**Function: `sendMessage(req, res)`**

- Parameters: `req.params.id` (conversation ID), `req.body` = `{ question }`, `req.user.id`
- Returns: SSE stream of `{ content }` events, final `{ done: true, matchedChunks }` event
- Flow:
  1. Validates `question` is present
  2. Finds conversation by ID AND userId
  3. Extracts `documentId` from `conversation.documentIds[0]` (currently only uses first document)
  4. Saves user message to DB
  5. Embeds the question via `embedText()`
  6. Runs vector search via `searchChunks(documentId, questionEmbedding)`
  7. Extracts citations (pageNumber, chunkIndex, score) from matched chunks
  8. Sets SSE headers (`Content-Type: text/event-stream`, `Cache-Control: no-cache`, `Connection: keep-alive`)
  9. Streams LLM response token-by-token via `streamAnswer()`
  10. Each token is written as `data: { content: "<token>" }\n\n`
  11. Saves complete assistant message with citations to DB
  12. Updates `conversation.updatedAt`
  13. Sends final `{ done: true, matchedChunks }` event
  14. Error handling: if headers not yet sent, returns JSON error; if mid-stream, sends error event

**Dependencies:**

- `../services/embedding.service` — `embedChunksForDocument` function

**Exports:** `{ uploadDocument, getDocumentFile, listDocuments, getDocument, updateDocument, trashDocument, restoreDocument, permanentDeleteDocument }`

**Function: `uploadDocument(req, res)`**

- Parameters: `req.file` (from multer), `req.user.id`
- Returns: `201` with `{ documentId, title, chunkCount, status }`
- Flow:
  1. Validates file was uploaded → `400` if not
  2. Creates Document record with `status: "processing"` — **warning:** this is outside the try/catch block; if `Document.create()` itself fails (e.g., DB connection issue), the error is unhandled and will crash the process
  3. Streams raw PDF buffer to GridFS bucket `"pdfs"` via `bucket.openUploadStream()`
  4. Captures GridFS file ID → `document.gridfsFileId`
  5. Extracts text from PDF buffer via `pdfParse(req.file.buffer)`
  6. Splits text by form-feed character (`\f`) into pages, filters empty pages
  7. For each page: cleans text → chunks text → creates chunk objects with `documentId`, `content`, `pageNumber`, `chunkIndex`
  8. Bulk inserts all chunks via `Chunk.insertMany()`
  9. Updates `document.pageCount`
  10. Calls `embedChunksForDocument(documentId)` to generate and store embeddings
  11. Sets `document.status = "ready"`
  12. On error: sets `document.status = "failed"`, returns `500`

**Function: `getDocumentFile(req, res)`**

- Parameters: `req.params.id` (document ID), `req.user.id`
- Returns: `200` with raw PDF binary stream, or `404` if not found
- Flow:
  1. Finds document by `_id` AND `userId` (ownership check)
  2. Returns `404` if not found
  3. Returns `404` with `"File not available"` if `gridfsFileId` is null (pre-GridFS documents)
  4. Opens GridFS download stream for `gridfsFileId`
  5. Sets `Content-Type: application/pdf` and `Content-Disposition: inline; filename="<title>"`
  6. Pipes GridFS stream directly to response
  7. On stream error: returns `404` with `"File not found"` (handles missing/corrupted GridFS files)

**Function: `listDocuments(req, res)`**

- Parameters: `req.query.filter` (optional: "recent", "starred", "trash"), `req.user.id`
- Returns: `200` with array of document objects
- Flow:
  - `"recent"` → non-trashed docs sorted by `lastOpenedAt` desc (nulls last), limit 20
  - `"starred"` → `isStarred: true, isTrashed: false`
  - `"trash"` → `isTrashed: true`
  - No filter → `isTrashed: false`, sorted by `createdAt` desc
  - All queries scoped to `userId: req.user.id`

**Function: `getDocument(req, res)`**

- Parameters: `req.params.id`, `req.user.id`
- Returns: `200` with document object
- Flow:
  1. Finds document by `_id` AND `userId` → `404` if not found
  2. Updates `lastOpenedAt` to `new Date()`
  3. Returns updated document

**Function: `updateDocument(req, res)`**

- Parameters: `req.params.id`, `req.body` = `{ title?, isStarred? }`, `req.user.id`
- Returns: `200` with updated document
- Flow:
  1. Builds partial update object from present body fields
  2. Uses `findOneAndUpdate` with ownership check → `404` if not found
  3. Returns updated document

**Function: `trashDocument(req, res)`**

- Parameters: `req.params.id`, `req.user.id`
- Returns: `200` with `{ message: "Document moved to trash" }`
- Flow:
  1. Finds document by `_id` AND `userId` → `404` if not found
  2. Sets `isTrashed: true`, `trashedAt: new Date()`
  3. Saves and returns confirmation message

**Function: `restoreDocument(req, res)`**

- Parameters: `req.params.id`, `req.user.id`
- Returns: `200` with updated document
- Flow:
  1. Finds document by `_id` AND `userId` → `404` if not found
  2. Sets `isTrashed: false`, `trashedAt: null`
  3. Saves and returns updated document

**Function: `permanentDeleteDocument(req, res)`**

- Parameters: `req.params.id`, `req.user.id`
- Returns: `200` with `{ message: "Document permanently deleted" }`
- Flow:
  1. Finds document by `_id` AND `userId` → `404` if not found
  2. Deletes all associated chunks via `Chunk.deleteMany({ documentId })`
  3. If `gridfsFileId` is set: attempts to delete GridFS file via `bucket.delete()` — wrapped in try/catch so a missing/corrupted GridFS file doesn't block the rest of the delete
  4. Deletes the Document record
  5. Returns confirmation message

### File: `backend/src/services/chunking.service.js`

**Purpose:** Splits cleaned text into overlapping chunks for embedding.

**Exports:** `{ cleanText, chunkText }`

**Constants:**

- `CHUNK_SIZE` = `600` (characters per chunk)
- `CHUNK_OVERLAP` = `100` (characters of overlap between consecutive chunks)

**Function: `cleanText(text)`**

- Parameters: raw page text string
- Returns: cleaned string
- Flow:
  1. Returns empty string if input is falsy
  2. Collapses multiple spaces/tabs into a single space
  3. Collapses 3+ consecutive newlines into 2
  4. Trims whitespace

**Function: `chunkText(text)`**

- Parameters: cleaned text string
- Returns: array of string chunks
- Flow:
  1. Returns empty array if input is falsy
  2. Slides a window of `CHUNK_SIZE` (600) characters with stride `CHUNK_SIZE - CHUNK_OVERLAP` (500) characters
  3. Each chunk is `text.slice(start, end)`
  4. Stops when start reaches or exceeds text length

**Chunking strategy:** Character-based fixed-size splitting with overlap. Each chunk is ~600 characters with ~100 characters of overlap with the next chunk to preserve context at boundaries.

### File: `backend/src/services/embedding.service.js`

**Purpose:** Generates vector embeddings for text using Google Gemini's embedding model.

**Dependencies:**

**Exports:** `{ embedText, embedChunksForDocument }`

**Function: `embedText(text)`**

- Parameters: text string to embed
- Returns: Promise resolving to array of 768 floats
- Flow:
  1. Gets generative model `gemini-embedding-001`
  2. Calls `model.embedContent()` with `outputDimensionality: 768`
  3. Returns `result.embedding.values`

**Function: `embedChunksForDocument(documentId)`**

- Parameters: MongoDB ObjectId of the document
- Returns: count of successfully embedded chunks (number)
- Flow:
  1. Finds all chunks for the given `documentId`
  2. Returns 0 if no chunks found

  4. For each batch: calls `embedText()` for each chunk in parallel via `Promise.allSettled()`
  5. Updates each chunk's `embedding` field in the database
  6. Logs rejected promises (individual failures)
  7. Waits 1 second between batches
  8. Throws `"Embedding failed for all chunks"` if 0 chunks were successfully embedded
  9. Returns total embedded count

### File: `backend/src/services/retrieval.service.js`

**Purpose:** Performs vector similarity search on document chunks using MongoDB Atlas Vector Search.

**Dependencies:**

- `mongoose` — For `mongoose.Types.ObjectId` conversion
- `../models/Chunk` — Chunk model for aggregation queries

**Exports:** `{ searchChunks }`

**Function: `normalizeForDedup(text)`**

**Function: `deduplicate(chunks)`**

- Parameters: array of chunk objects
- Returns: deduplicated array
- Logic: For each chunk, checks if any previously seen chunk's normalized content contains the current chunk's normalized content (or vice versa) with a length ratio > 0.6. If so, it's considered a duplicate and skipped.

**Function: `searchChunks(documentId, questionEmbedding, topK = 5)`**

  2. Deduplicates the results
  3. Returns final array

## 7. Backend Logic Flow

### Flow 1: User Signup

1. `POST /api/auth/signup` received at `auth.routes.js`
2. No middleware (public route)
3. `auth.controller.js:signup()` executes
4. Validates `name`, `email`, `password` are present → `400` if missing
5. Validates email format with regex → `400` if invalid
6. Validates password length >= 6 → `400` if too short
7. Queries `users` collection for existing email → `409` if found
8. Calls `auth.service.js:hashPassword(password)` → bcrypt with 10 salt rounds
9. Creates user document in `users` collection
10. Calls `auth.service.js:generateToken(user._id)` → signs JWT with 7-day expiry
11. Returns `201` with token and user info (no passwordHash)

### Flow 2: User Login

1. `POST /api/auth/login` received at `auth.routes.js`
2. `auth.controller.js:login()` executes
3. Validates `email` and `password` are present → `400` if missing
4. Queries `users` collection for email → `401` if not found
5. Calls `auth.service.js:comparePassword(password, user.passwordHash)` → bcrypt compare
6. Returns `401` if no match
7. Calls `auth.service.js:generateToken(user._id)`
8. Returns `200` with token and user info

### Flow 3: PDF Upload → GridFS → Chunk → Embed

1. `POST /api/documents` received at `document.routes.js`
2. `auth.middleware.js` verifies JWT → attaches `req.user`
3. `upload.middleware.js` (multer) parses multipart form-data → attaches `req.file`
4. `document.controller.js:uploadDocument()` executes
5. Validates `req.file` exists → `400` if not
6. Creates Document record with `status: "processing"`
7. Streams raw PDF buffer to GridFS bucket `"pdfs"` via `bucket.openUploadStream()`
8. Captures upload stream `_id` → saves as `document.gridfsFileId`
9. `pdf-parse` extracts text from `req.file.buffer`
10. Splits text by `\f` (form-feed) into pages, filters empty pages
11. For each page (1-indexed):
    - `chunking.service.js:cleanText(pageText)` → normalizes whitespace
    - `chunking.service.js:chunkText(cleaned)` → splits into ~600-char chunks with 100-char overlap
    - Creates chunk objects with `documentId`, `content`, `pageNumber`, `chunkIndex`
12. `Chunk.insertMany(allChunks)` → bulk inserts all chunks
13. Updates `document.pageCount`
14. `embedding.service.js:embedChunksForDocument(documentId)`:
    - Finds all chunks for this document
    - Processes in batches of 8:
      - For each chunk: calls Gemini `gemini-embedding-001` → 768-dim vector
      - Updates chunk document with embedding
    - Waits 1s between batches
    - Throws if ALL chunks fail
15. Sets `document.status = "ready"`
16. Returns `201` with `{ documentId, title, chunkCount, status }`
17. On any error: sets `document.status = "failed"`, returns `500`

### Flow 4: Ask a Question (Chat)

1. `POST /api/conversations/:id/messages` received at `conversation.routes.js`
2. `auth.middleware.js` verifies JWT → attaches `req.user`
3. `conversation.controller.js:sendMessage()` executes
4. Validates `question` is present → `400` if missing
5. Finds conversation by `_id` AND `userId` → `404` if not found (ownership check)
6. Gets `documentId` from `conversation.documentIds[0]`
7. Saves user message to `messages` collection
8. `embedding.service.js:embedText(question)` → Gemini → 768-dim question embedding
9. `retrieval.service.js:searchChunks(documentId, questionEmbedding)`:
   - Runs `$vectorSearch` aggregation on `chunks` collection
   - Filtered by `documentId`, top 5, numCandidates 100
   - Filters by score >= 0.75
   - Deduplicates near-identical chunks
10. Extracts citations from matched chunks
11. Sets SSE headers on response
12. `llm.service.js:streamAnswer(question, matchedChunks)`:
    - Builds prompt: system instructions + formatted context chunks + question
    - Calls Groq `llama-3.3-70b-versatile` with `stream: true`
    - Yields tokens as they arrive
13. Writes each token as SSE `data: { content }` event
14. After stream completes: saves assistant message with citations to `messages` collection
15. Updates `conversation.updatedAt`
16. Sends final `data: { done: true, matchedChunks }` event
17. Calls `res.end()`

### Flow 5: Download PDF from GridFS

1. `GET /api/documents/:id/file` received at `document.routes.js`
2. `auth.middleware.js` verifies JWT → attaches `req.user`
3. `document.controller.js:getDocumentFile()` executes
4. Finds document by `_id` AND `userId` → `404` if not found
5. Checks `gridfsFileId` is not null → `404` with `"File not available"` if null
6. Opens GridFS download stream for `gridfsFileId`
7. Sets `Content-Type: application/pdf` and `Content-Disposition: inline; filename="<title>"`
8. Pipes GridFS stream to `res`
9. On stream error (file missing/corrupted): returns `404`

### Flow 6: Permanent Delete (with GridFS cleanup)

1. `DELETE /api/documents/:id/permanent` received at `document.routes.js`
2. `auth.middleware.js` verifies JWT → attaches `req.user`
3. `document.controller.js:permanentDeleteDocument()` executes
4. Finds document by `_id` AND `userId` → `404` if not found
5. Deletes all chunks: `Chunk.deleteMany({ documentId })`
6. If `gridfsFileId` is set: attempts `bucket.delete(gridfsFileId)` — wrapped in try/catch so a missing file doesn't block deletion
7. Deletes the Document record: `Document.deleteOne({ _id })`
8. Returns `200` with `{ message: "Document permanently deleted" }`

## 11. Important Development Rules

### Architecture Rules

1. **Layered structure:** Always follow routes → controllers → services → models. Never put business logic in routes or controllers. Controllers parse requests and send responses; services contain logic.

2. **`dotenv.config()` must stay as the first line in `server.js`** — before any other `require()`. Modules like `config/gemini.js` and `config/groq.js` read `process.env` at import time.

3. **Every protected route must use `auth.middleware.js`** — it attaches `req.user.id` from the verified JWT.

4. **All Mongo queries on user-owned data must filter by `userId`** — every route fetching a specific document, chunk, conversation, or message by ID must include `userId: req.user.id` in the same query. This is the only authorization enforcement (MongoDB has no row-level security).

5. **Never return `passwordHash`** or any sensitive field in API responses. Use `.select("-passwordHash")` or explicitly shape the response object.

6. **Async/await throughout** — no raw `.then()` chains.

7. **Match existing file/folder patterns** when adding new features. Don't introduce new structural patterns without a reason.

### Code Patterns

8. **Error handling in controllers:** Use try/catch. Return deliberate status codes directly (400 validation, 401 auth, 404 not found, 409 conflict). Unexpected errors return 500 with `{ message: error.message }`. Alternatively, errors can be passed to `next(err)` for the centralized `error.middleware.js` to handle.

9. **`Promise.allSettled` for batch operations** — log rejections individually and throw if ALL fail. Don't silently swallow errors.

10. **`pdf-parse` must stay at version `1.1.1`** — v2.x completely changed its export shape (exports a class instead of a function). Do not upgrade without rewriting extraction code.

11. **Multer uses memory storage** — files are in `req.file.buffer`, not on disk. This is intentional for the current architecture.

12. **SSE streaming pattern** — when streaming, set headers first (`Content-Type: text/event-stream`, etc.), then write `data: <json>\n\n` events, then `res.end()`. Handle errors mid-stream by writing error events.

13. **GridFS lazy init** — `config/gridfs.js` exports a factory function, not an instance. Call `getGridFSBucket()` only after `connectDB()` has succeeded. Never call it at module import time.

14. **GridFS cleanup is best-effort** — in `permanentDeleteDocument`, GridFS deletion is wrapped in its own try/catch. A missing/corrupted GridFS file should not block Document + Chunks deletion.

### Files Not to Modify Without Care

15. **`server.js`** — the order of `dotenv.config()` and other imports is critical
16. **`config/gemini.js`** and **`config/groq.js`** — these read env vars at import time
17. **`models/Chunk.js`** — the `embedding` field structure must match the Atlas Vector Search index (768 dimensions, cosine similarity)
18. **Atlas Vector Search index** — created in Atlas UI, not in code. Changing the schema without updating the index will break retrieval
19. **`config/gridfs.js`** — bucket name and connection dependency
